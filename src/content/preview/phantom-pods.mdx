I run the stateless workload platform for my company. One day, I needed to add an annotation to a workload's pods. Pure metadata, the Kubernetes equivalent of sticking a Post-It note on a resource. Patched the change. The controller reconciled. And new pods started spinning up.

But, why?

## Something's Off

We use [OpenKruise](https://github.com/openkruise/kruise) **CloneSet** to schedule stateless workloads. CloneSet supports **in-place updates**: patching pod metadata without deleting the pod. No restart, no rescheduling. An annotation change should be a no-op.

But right after we pushed the change, about 5% of pods were being *recreated*: new pods spun up, old pods terminated. In-place update means zero new pods. So where were they coming from?

## Two Knobs

Before we dig into code, one concept matters. CloneSet rolling updates are governed by two knobs: `maxUnavailable` (how many pods can be down at once) and `maxSurge` (how many extra pods to create above the desired count). Together they set the rollout pace. Step through a 100-replica rollout with maxSurge at 10% and maxUnavailable at 5%:

<RollingUpdateVisualizer replicas={100} maxSurge={10} maxUnavailable={5} />

maxSurge makes rolling updates fast and safe: new pods come up before old ones go down. But surge exists to *replace* pods. What happens when pods do not need replacing? Our CloneSet had maxSurge set to 5%, and exactly 5% of pods were being phantom-created during an in-place update. Not a coincidence.

This function computes the delta between current state and desired state: how many pods to create, delete, update in place.

Deep inside sits the surge logic:

It sees a revision diff, computes surge, and tells the controller to create pods. Every time. It does not check whether the update can happen in place. That decision lives downstream. The surge calculation is blind to the update strategy.

## The Wrong Order

The reconciliation pipeline runs in a fixed order: count how many pods need updating, compute surge, create the surge pods, *then* check for in-place updates, and finally delete surplus. For recreate updates, this ordering works. But for in-place updates, surge computation and pod creation fire before the controller knows that patching will handle the change:

Callers pre-compute the boolean via `CanUpdateInPlace()`. The function stays pure. All existing tests pass unchanged: `canInPlaceUpdate=false` preserves the original behavior.

I opened a PR to fix this upstream in OpenKruise: [openkruise/kruise#2377](https://github.com/openkruise/kruise/pull/2377).

## Takeaways

Ordering matters in the control loop. Surge calculation runs before the controller checks whether in-place update will handle the change. Each feature works fine alone: in-place update with `maxSurge=0` never triggers it, and surge with recreate updates behaves as designed. The bug only shows up when both are active, an intersection no single-feature test covered.

The fix adds one boolean gating the surge path. Every cluster running `InPlaceIfPossible` with `maxSurge > 0` was silently hitting this.
