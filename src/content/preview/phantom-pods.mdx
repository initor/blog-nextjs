You are running the workload platform for the company. One day you need to add an annotation to a fleet of pods. Pure metadata, the Kubernetes equivalent of sticking a Post-It note on a resource.

You push the change. The controller reconciles. And...

Wait. Why are new pods spinning up?

## Something's Off

We use [OpenKruise](https://github.com/openkruise/kruise) **CloneSet** to schedule stateless workloads. CloneSet supports **in-place updates**: patching pod metadata without deleting the pod. No restart, no rescheduling. An annotation change should be a no-op.

But right after we pushed the change, about 5% of pods were being *recreated*: new pods spun up, old pods terminated. In-place update means zero new pods. So where were they coming from?

## Two Knobs

Before we dig into code, one concept matters: how CloneSet rolls out changes.

Two knobs control the rollout:

- `maxUnavailable`: how many pods can be down at once. The safety limit.
- `maxSurge`: how many *extra* pods to create above the desired count. The speed lever.

Together they set the pace. Here is what that looks like with 100 replicas, maxSurge at 10%, and maxUnavailable at 5%, and step through the same rollout interactively:

<RollingUpdateVisualizer replicas={100} maxSurge={10} maxUnavailable={5} />

maxSurge makes rolling updates fast and safe: new pods come up before old ones go down. Capacity stays healthy throughout.

But here is the thing. Surge exists to *replace* pods. What happens when pods do not need replacing?

Our CloneSet had maxSurge set to 5%. And exactly 5% of pods were being phantom-created during an in-place update. Not a coincidence.

This function computes the delta between current state and desired state: how many pods to create, delete, update in place.

Deep inside sits the surge logic:

It sees a revision diff, computes surge, and tells the controller to create pods. Every time. It does not check whether the update can happen in place. That decision lives downstream. The surge calculation is blind to the update strategy.

## One Boolean

The fix is a single guard:

```go

    ...
    canInPlaceUpdate bool,          // new
    isPodUpdate      IsPodUpdateFunc,
) (res expectationDiffs) {
    // ...
    // When in-place update is possible, surge is unnecessary.
    if !canInPlaceUpdate && util.IsIntPlusAndMinus(updateOldDiff, updateNewDiff) {
    // ^^^^^^^^^^^^^^^^^    one boolean gate
```

Callers pre-compute the boolean via `CanUpdateInPlace()`. The function stays pure. All existing tests pass unchanged: `canInPlaceUpdate=false` preserves the original behavior.

I opened a PR to fix this upstream in OpenKruise: [openkruise/kruise#2377](https://github.com/openkruise/kruise/pull/2377).

## Takeaways

The root cause was ordering. Surge calculation runs before the controller checks whether in-place update will handle the change. Each feature works fine alone: in-place update with `maxSurge=0` never triggers it, and surge with recreate updates behaves as designed. The bug only shows up when both are active, an intersection no single-feature test covered.

The fix adds one boolean gating the surge path. Every cluster running `InPlaceIfPossible` with `maxSurge > 0` was silently hitting this.
