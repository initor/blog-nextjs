Over three days, four of us shipped ten production-ready agent skills covering a workload platform end-to-end. One agent, ten structured specifications, zero lines of hand-written code. The kind of output that used to take a team weeks. But the real discovery was not the speed. It was what happened after the weekend: the agents could build, yet they could not sustain that ability across sessions. Every time a session ended, everything the agent had learned vanished. The scarce resource was never intelligence or execution. It was the architecture for persisting state so that work could compound instead of restart.

## The Hackathon

If you run a platform serving thousands of engineers, you know what operational toil feels like. Failed deploys, throttled services, hardware remediations, capacity rebalancing across pools. Every incident lands on a platform engineer who has to investigate, trace logs, and respond with judgment. When the platform spans hundreds of thousands of machines, toil is not a side effect of poor tooling. It is inherent to scale.

We had tried to automate this three times before the hackathon. A Slackbot, a chatbot wrapper, an MCP server. Different interfaces, same flaw: without structured context modeling, every interaction collapsed to a single-round LLM call. Answers were generic because the tools had no curated knowledge of each problem domain. The fourth attempt changed the approach entirely. Before the hackathon, I used ChatGPT to design a complete protocol for agent-driven development: not a template, but a structured path from skill initialization through testing and verification.

Three days, four engineers, ten skills. The agent read each structured specification, built the implementation, wrote the tests, committed the result. The human wrote specifications; the agent wrote code. That inversion worked. But it also created a new problem: managing context across sessions, so that work spanning days or weeks could stop and resume without losing its place.

## The Session Problem

The hackathon worked. But it also surfaced a problem I had not anticipated.

When you close an agent session and open a new one, everything is gone. The context window is ephemeral: whatever the previous session learned, decided, or built is invisible to the next. During the hackathon this was manageable because each skill fit inside a single session. But real projects span days or weeks. The first session makes progress. The second starts from scratch.

An agent with no memory is not a collaborator. It is a stranger you have to re-brief every morning.

The shared understanding of what has been tried, what has been decided, what comes next: all of it vanishes the moment you hit "new session." The hackathon proved that agents could execute. The question now was whether they could sustain execution across time.

The solution, when it emerged, was almost embarrassingly simple. If the context window is ephemeral, externalize the state into files that persist. Not documentation written for humans, but memory written for agents:

- **ROADMAP.md** tracks milestones and phases
- **STATE.md** records the current position: which phase, which plan, what was last accomplished
- **CONTEXT.md** captures decisions and their rationale
- **PLAN.md** breaks the current phase into concrete tasks
- **SUMMARY.md** distills what each session accomplished and what comes next

These files are not bureaucratic overhead. They are the mechanism by which one session hands off to the next. The agent writes a summary of what it accomplished and what remains; the next session reads that summary and picks up exactly where the previous one left off.

What surprised me was how quickly this compounded. By the fifth phase of a project, the agent already knew decisions from phases one through four. Not because it remembered them, since each session's context window started empty, but because the planning files carried forward a distilled version of everything that mattered. Irrelevant details were pruned. Dead-end explorations were dropped. Key decisions and their reasoning were preserved. Each new session inherited not the full history, but the right history: the subset that enabled it to continue with full situational awareness.

*Each session starts with a narrower, more precise context than the last. Threads that led nowhere are pruned. Threads that matter are summarized. The agent does not remember prior sessions; it reads their distilled output and picks up with the decisions already made.*

The agent was not starting over each time. It was starting ahead, with the accumulated judgment of every prior session compressed into a few files it could load in seconds.

The context boundary was not the conversation window. It was the working directory.

That insight unlocked something I had not planned for. After the hackathon, I kept using the same pattern for projects that had nothing to do with coding. Engineering design documents that took a week of research and drafting. Document reviews where I needed to track feedback across multiple revision cycles. Prototyping sessions where ideas evolved over days. Every one of these projects had the same structural requirement: work that spanned multiple sessions, where continuity between sessions determined quality. The working directory became the universal context boundary. Any directory could be a project, and any project could carry its own planning files, its own state, its own accumulated decisions.

*Each project directory carries its own context boundary. Planning, state, and instructions are scoped to a single directory, whether the project is a codebase, a design document, or a prototype. The pattern is the same; only the artifacts differ.*

The manual version of this workflow was effective but tedious. During the hackathon, ChatGPT had helped me design the protocols for structuring each skill specification. Afterward, I applied the same approach to context management: using ChatGPT to draft the markdown files, the project goals, the milestone tracking, the pause and resume protocols. It worked, but I was still the human in the loop, repetitively prompting for each new project, copying templates, initializing the same file structures by hand. The insight had become a pattern, but the pattern had not yet become infrastructure.

Then I found [GSD (get-shit-done)](https://github.com/gsd-build/get-shit-done), a Claude Code plugin that automated the entire lifecycle. What I had been doing manually, bootstrapping a project directory with planning files, breaking work into phases, tracking progress across sessions, verifying completion, GSD handled automatically. The evolution was clean: manual context management during the hackathon, semi-automated protocols designed with ChatGPT's help for the projects that followed, and finally a fully automated plugin that turned the pattern into a single command. The insight from the hackathon, that agents need externalized state to work across sessions, had become infrastructure that any project could use from the first minute.

## Conclusion

It wasn't always this way.

For most of my career, the bottleneck was knowledge. You learned by reading papers, studying source code, and accumulating experience through years of trial and error. In 2014 I spent a semester trying to understand Raft consensus, one paper a week, re-reading sections until the mental model held. Getting good at something meant putting in the time. The gap between a junior and a senior was measured in calendar years, not talent.

Then, around 2023, chatbots collapsed the acquisition cost. You could ask a synthesized question and get a useful answer in seconds instead of hours spent tracing through documentation. Learning got faster, but the execution model didn't change. You still wrote the code, ran the tests, debugged the failures. The bottleneck moved from "I don't know how" to "I know how but haven't done it yet." A faster reference manual, but the work was still yours to do.

Now the bottleneck is neither knowledge nor execution. It is sustaining context across sessions. The hackathon proved that agents can build production software from structured specifications. But every project longer than a single session faces the same structural problem: the context window is ephemeral, and without externalized state, each new session starts from zero. The scarce resource is not engineering hours or clarity of intent. It is the architecture that lets work compound, planning files that carry forward what was decided, summaries that compress what matters, state that tells the next session exactly where to pick up. Get that right and a weekend of building becomes a month of building without losing a step.
