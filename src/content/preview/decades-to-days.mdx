Over three days, four of us, working half-time between other commitments, shipped ten production-ready agent skills covering a workload platform end-to-end: deployment pipelines, hardware remediation, disruption management, cluster operations. One agent, ten structured specifications, zero lines of hand-written code. The kind of output that used to take a team weeks of design, implementation, and testing. That weekend collapsed decades of learning-to-build into days of building. The question worth asking is: what changed?

## The Hackathon

If you run a platform serving thousands of engineers, you know what operational toil feels like before you can put numbers on it. Someone's deploy failed and they need to know why. A service is being throttled and the owner wants an explanation. Behind the scenes, hardware is being remediated, nodes are draining, and capacity is being rebalanced across pools. Every one of these situations lands on a platform engineer who has to investigate the specifics, trace through logs, and respond with judgment. The volume is relentless. When the platform spans hundreds of thousands of machines, thousands of repos, and tens of thousands of daily users, toil is not a side effect of poor tooling. It is inherent to platform ownership at scale.

This is the problem that motivated the hackathon. But we had already tried three times before. A Slackbot that queried platform APIs. A chatbot wrapper with function calling over the same APIs. An MCP server exposing tool endpoints. All three shared the same flaw: they lacked structured context modeling, which reduced every interaction to a single-round LLM call. Without curated base context specific to each question domain, answers were generic. Without a structured way to model agents and skills, every attempt was essentially "just chat." The interface changed each time; the fundamental limitation never did.

The fourth attempt changed the approach entirely. Before the hackathon, I used ChatGPT to design the full end-to-end workflow protocol: how to initialize a skill, what sections each specification should include, how to test each skill, how to iterate on failures. Not a template, but a complete protocol for structuring agent-driven development. At the time, the core bottleneck was manual context management across multi-session projects. Each phase required tracking progress and preparing prompts without losing prior context. Too much human in the loop, too error-prone. ChatGPT's workflow protocol addressed this by giving the executing agent (Claude Code) a structured path from initialization through verification.

Three days, four engineers, ten skills. The skills covered workload scheduling failures, data-plane runtime failures, planned disruptions, application log retrieval, and pool capacity analysis. The speed made the breadth possible: because each skill took hours rather than weeks, ten operational domains could be covered in a single weekend. The agent read each structured specification, built the implementation, wrote the tests, committed the result. The human described what was needed; the agent implemented it. That inversion of the traditional relationship, from writing code to writing specifications, worked. But it also created a new problem: managing context across sessions, so that work spanning days or weeks could stop and resume without losing its place.

## The Journey

After the hackathon, I continued with personal agentic development, not limited to coding. Engineering doc authoring, doc reviewing, prototyping ideas. These are all projects that span days or weeks. No single LLM session can finish them. I needed systematic context summarization and pruning between sessions: base context setup, phase and milestone planning, completion verification, progress tracking so you can stop and resume without losing your place.

I started using ChatGPT to set up the markdown files: project goals, base context, milestone and phase tracking, pause and resume protocols. This was human-in-the-loop; I still had to repetitively prompt for each new project. Then I discovered [GSD (get-shit-done)](https://github.com/gsd-build/get-shit-done), a Claude Code plugin that automated the bootstrap entirely.

Previously, my working directory was always a code repository. Now it's the working directory for any project: doc authoring, doc reviewing, prototyping. I initialize GSD in every directory that is a project root. Everything is tracked by version control: artifacts plus progress status in structured markdown files. Each directory is a self-contained context boundary.

## Conclusion

It wasn't always this way.

For most of my career, the bottleneck was simply knowledge. You learned by reading papers, studying source code, and accumulating experience through years of trial and error. In 2014 I spent a semester just trying to understand Raft consensus, one paper a week, re-reading sections until the mental model held. Getting good at something meant putting in the time, and the gap between a junior and a senior was measured in calendar time, not talent.

Then, around 2023, chatbots loosened the acquisition cost. You could ask a synthesized question and get a useful answer in seconds instead of hours of deep-diving through documentation. Learning got faster, but the execution model didn't change. You still wrote the code, ran the tests, debugged the failures. The bottleneck had moved from "I don't know how" to "I know how but haven't done it yet." A faster reference manual, but the work was still yours to do.

Now the bottleneck has shifted upstream of both knowledge and execution. It's about orchestrating context: defining clear boundaries, writing precise specs, and structuring work so that agents can execute independently. The scarce resource is no longer engineering hours. It's clarity of intent.
