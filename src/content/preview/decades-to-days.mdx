Over three days, four of us, working half-time between other commitments, shipped ten production-ready agent skills covering a workload platform end-to-end: deployment pipelines, hardware remediation, disruption management, cluster operations. One agent, ten structured specifications, zero lines of hand-written code. The kind of output that used to take a team weeks of design, implementation, and testing. That weekend collapsed decades of learning-to-build into days of building. The question worth asking is: what changed?

## The Hackathon

If you run a platform serving thousands of engineers, you know what operational toil feels like before you can put numbers on it. Someone's deploy failed and they need to know why. A service is being throttled and the owner wants an explanation. Behind the scenes, hardware is being remediated, nodes are draining, and capacity is being rebalanced across pools. Every one of these situations lands on a platform engineer who has to investigate the specifics, trace through logs, and respond with judgment. The volume is relentless. When the platform spans hundreds of thousands of machines, thousands of repos, and tens of thousands of daily users, toil is not a side effect of poor tooling. It is inherent to platform ownership at scale.

This is the problem that motivated the hackathon. But we had already tried three times before. A Slackbot that queried platform APIs. A chatbot wrapper with function calling over the same APIs. An MCP server exposing tool endpoints. All three shared the same flaw: they lacked structured context modeling, which reduced every interaction to a single-round LLM call. Without curated base context specific to each question domain, answers were generic. Without a structured way to model agents and skills, every attempt was essentially "just chat." The interface changed each time; the fundamental limitation never did.

The fourth attempt changed the approach entirely. Before the hackathon, I used ChatGPT to design the full end-to-end workflow protocol: how to initialize a skill, what sections each specification should include, how to test each skill, how to iterate on failures. Not a template, but a complete protocol for structuring agent-driven development. At the time, the core bottleneck was manual context management across multi-session projects. Each phase required tracking progress and preparing prompts without losing prior context. Too much human in the loop, too error-prone. ChatGPT's workflow protocol addressed this by giving the executing agent (Claude Code) a structured path from initialization through verification.

Three days, four engineers, ten skills. The skills covered workload scheduling failures, data-plane runtime failures, planned disruptions, application log retrieval, and pool capacity analysis. The speed made the breadth possible: because each skill took hours rather than weeks, ten operational domains could be covered in a single weekend. The agent read each structured specification, built the implementation, wrote the tests, committed the result. The human described what was needed; the agent implemented it. That inversion of the traditional relationship, from writing code to writing specifications, worked. But it also created a new problem: managing context across sessions, so that work spanning days or weeks could stop and resume without losing its place.

## The Session Problem

The hackathon worked. But it also surfaced a problem I had not anticipated.

When you close an agent session and open a new one, everything is gone. The context window is ephemeral. Whatever the previous session learned, whatever it decided, whatever it built, all of it is invisible to the next session. During the hackathon this was manageable because each skill fit inside a single session. But any project that spans days or weeks, and most real projects do, requires multiple sessions. The first session makes progress. The second session starts from scratch.

This is not a minor inconvenience. It is a structural limitation. An agent with no memory of prior sessions is not a collaborator; it is a stranger you have to re-brief every morning. The context that makes collaboration productive, the shared understanding of what has been tried, what has been decided, what comes next, vanishes the moment you hit "new session." I realized that the bottleneck had shifted. The hackathon proved that agents could execute. The question now was whether they could sustain execution across time.

The solution, when it emerged, was almost embarrassingly simple. If the context window is ephemeral, externalize the state into files that persist. Not documentation written for humans, but memory written for agents. A ROADMAP.md that tracks milestones and phases. A STATE.md that records the current position: which phase, which plan, what was last accomplished. A CONTEXT.md that captures decisions and their rationale. A PLAN.md that breaks the current phase into concrete tasks. A SUMMARY.md that distills what each session accomplished and what comes next. These files are not bureaucratic overhead. They are the mechanism by which one session hands off to the next. The agent writes a summary of what it accomplished and what remains; the next session reads that summary and picks up exactly where the previous one left off.

What surprised me was how quickly this compounded. By the fifth phase of a project, the agent already knew decisions from phases one through four. Not because it remembered them, since each session's context window started empty, but because the planning files carried forward a distilled version of everything that mattered. Irrelevant details were pruned. Dead-end explorations were dropped. Key decisions and their reasoning were preserved. Each new session inherited not the full history, but the right history: the subset that enabled it to continue with full situational awareness.

This pattern of distilling what matters between sessions changed how I thought about multi-session work. The agent was not starting over each time. It was starting ahead, with the accumulated judgment of every prior session compressed into a few files it could load in seconds. The context boundary was not the conversation window. It was the working directory.

That insight unlocked something I had not planned for. After the hackathon, I kept using the same pattern for projects that had nothing to do with coding. Engineering design documents that took a week of research and drafting. Document reviews where I needed to track feedback across multiple revision cycles. Prototyping sessions where ideas evolved over days. Every one of these projects had the same structural requirement: work that spanned multiple sessions, where continuity between sessions determined quality. The working directory became the universal context boundary. Any directory could be a project, and any project could carry its own planning files, its own state, its own accumulated decisions.

The manual version of this workflow was effective but tedious. During the hackathon, ChatGPT had helped me design the protocols for structuring each skill specification. Afterward, I applied the same approach to context management: using ChatGPT to draft the markdown files, the project goals, the milestone tracking, the pause and resume protocols. It worked, but I was still the human in the loop, repetitively prompting for each new project, copying templates, initializing the same file structures by hand. The insight had become a pattern, but the pattern had not yet become infrastructure.

Then I found [GSD (get-shit-done)](https://github.com/gsd-build/get-shit-done), a Claude Code plugin that automated the entire lifecycle. What I had been doing manually, bootstrapping a project directory with planning files, breaking work into phases, tracking progress across sessions, verifying completion, GSD handled automatically. The evolution was clean: manual context management during the hackathon, semi-automated protocols designed with ChatGPT's help for the projects that followed, and finally a fully automated plugin that turned the pattern into a single command. The insight from the hackathon, that agents need externalized state to work across sessions, had become infrastructure that any project could use from the first minute.

## Conclusion

It wasn't always this way.

For most of my career, the bottleneck was simply knowledge. You learned by reading papers, studying source code, and accumulating experience through years of trial and error. In 2014 I spent a semester just trying to understand Raft consensus, one paper a week, re-reading sections until the mental model held. Getting good at something meant putting in the time, and the gap between a junior and a senior was measured in calendar time, not talent.

Then, around 2023, chatbots loosened the acquisition cost. You could ask a synthesized question and get a useful answer in seconds instead of hours of deep-diving through documentation. Learning got faster, but the execution model didn't change. You still wrote the code, ran the tests, debugged the failures. The bottleneck had moved from "I don't know how" to "I know how but haven't done it yet." A faster reference manual, but the work was still yours to do.

Now the bottleneck has shifted upstream of both knowledge and execution. It's about orchestrating context: defining clear boundaries, writing precise specs, and structuring work so that agents can execute independently. The scarce resource is no longer engineering hours. It's clarity of intent.
